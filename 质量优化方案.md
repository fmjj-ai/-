# 质量优化方案

## 1. 文档目标

本文档聚焦“如何把项目从能运行提升到高质量可长期使用”。优化对象包括性能、稳定性、正确性、复现性、可维护性、可观测性、安全性与体验一致性。

## 2. 质量目标

- 不做只跑主流程的脆弱系统
- 不做只能演示一次的样板系统
- 不做无法复现结果的分析系统
- 不做越用越卡、越用越乱的系统

最终目标是：

- 可长期自用
- 可持续扩展
- 结果可信
- 出错可追
- 数据可回滚
- 功能可审查

## 3. 优化优先级

### P0 必做

- 快照与工作副本隔离
- 任务系统完整
- 错误结构统一
- 图表与表格性能兜底
- 导出中心与产物追踪
- 日志与审计

### P1 应做

- 缓存命中
- 主题系统
- 模板复用
- 只读模式
- 生命周期清理

### P2 可迭代增强

- 更细的质量规则引擎
- 更高级的实验对比能力
- 更完整的报告智能解读

## 4. 重点优化项

### 4.1 数据存储分层

原因：

- SQLite 不适合直接承担百万级工作表高频读写。

影响：

- 提升大数据读写性能
- 降低数据库膨胀风险
- 方便做快照与缓存

回滚方式：

- 若实现复杂度过高，可先以 CSV/Parquet 双格式落盘，保留元数据映射不变

### 4.2 任务系统一体化

原因：

- 导入、建模、主题分析、导出等操作耗时长，若无统一任务系统，会出现状态丢失、取消无效、日志缺失。

影响：

- 可观测性显著提升
- 异常排查成本下降

回滚方式：

- 若某些短任务先不同步走 Task，可保留同步接口，但必须限制在亚秒级操作

### 4.3 图表性能与资源释放

原因：

- 图表是最容易造成前端卡顿和 GPU 内存泄漏的区域。

影响：

- 页面稳定性提升
- 多图切换时不卡死

回滚方式：

- 若某些图表类型难以支持双模式导出，先确保 Canvas 交互稳定，再逐个补 SVG 导出

### 4.4 表格性能

原因：

- 大表如果由前端全量持有，会直接拖垮交互。

影响：

- 百万级数据也能可控浏览

回滚方式：

- 如果暂时无法做复杂筛选，可先支持基础关键字检索，但不能退回全量加载

### 4.5 可复现性强化

原因：

- 分析系统若不能复现，输出价值会快速下降。

影响：

- 结果可信
- 迭代成本降低

回滚方式：

- 不允许取消该能力，只允许先做最小集后扩展

### 4.6 安全与风险提示

原因：

- DeepSeek 等外部 API 涉及成本与数据外发风险。

影响：

- 降低误操作与泄密风险

回滚方式：

- 若外部 API 方案尚未完成，必须明确降级为本地模式，不允许伪装成已支持

## 5. 质量度量

建议建立以下质量指标：

- 需求覆盖率
- 审查闭环率
- 测试通过率
- P0/P1 缺陷数量
- 缓存命中率
- 任务失败率
- 导出成功率
- SSE 断连恢复成功率

## 6. 可维护性优化

### 前端

- 组件按领域拆分
- 页面避免巨型单文件
- store 职责单一
- 图表配置抽象为 schema

### 后端

- API 层和服务层解耦
- 算法能力独立服务化
- 任务逻辑与 HTTP 路由分离
- 仓储层统一数据访问

## 7. 稳定性优化

- 导入前先校验格式与编码
- 大任务分阶段写进度
- 任务异常自动写日志和错误建议
- 导出失败不应污染产物中心
- 删除操作必须软保护

## 8. 用户体验优化

- 首页保留最近使用与失败摘要
- 工作台顶部持续显示当前项目与快照
- 任务完成后可一键跳转结果
- 错误提示直接给下一步建议

## 9. 质量优化执行顺序

- 第一阶段：快照、Run、Task、Artifact 四件套打牢
- 第二阶段：表格、图表、导出性能兜底
- 第三阶段：错误结构、日志、通知打通
- 第四阶段：缓存、生命周期、只读模式、会话保护
- 第五阶段：细节优化、风格统一、剩余缺陷清零

## 10. 完成标准

质量优化子 agent 最终必须给出：

- 优化项清单
- 每项的原因、影响、回滚方式
- 已完成项
- 未完成项
- 剩余风险

通过标准：

- 系统在高频使用场景下仍稳定
- 没有明显的性能硬伤
- 没有会导致数据错误或不可复现的结构缺陷
