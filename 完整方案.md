# 自用数据统计分析系统完整方案

## 1. 文档目标

本文档用于把 [需求文档_v2.md](D:\数据分析系统\需求文档_v2.md) 拆解为一套可执行的落地方案，供主 agent 与子 agent 协同实现。本文档只基于需求文档，不参考现有代码。

目标有四个：

- 把需求转成可落地的系统架构与实施路径。
- 明确前后端、任务系统、数据存储、导出中心之间的边界。
- 给后续前端、后端、联调、审查、测试、质量优化工作提供统一总纲。
- 强化“质量优先、可复现、可迭代、可审计”的实现原则，避免只做表面可跑通的流程。

## 2. 建设原则

### 2.1 第一原则

- 质量优先，不接受“为了先跑通而牺牲正确性、可维护性、可复现性”的实现。
- 所有长任务必须可追踪、可取消、可重试。
- 所有关键结果必须可导出、可复现、可追溯来源。
- 所有危险操作必须二次确认，并说明影响范围。
- 所有功能必须围绕“单人长期自用、持续积累、重复复用”设计。

### 2.2 技术原则

- 前端负责交互、配置、可视化和局部异步计算，不承担大数据主计算职责。
- 后端负责数据导入、清洗、统计、建模、任务调度、产物管理和持久化。
- 元数据与大数据实体分离存储。
- 任务状态、运行记录、快照版本必须结构化保存。
- 接口统一返回结构，错误必须可理解、可行动。

### 2.3 质量门槛

- 需求覆盖率不能低于 95%，剩余项必须有明确占位与后续计划。
- 至少完成 5 轮审查和修复闭环。
- 核心闭环必须通过：
  - 导入 -> 统计 -> 出图 -> 导出
  - 导入 -> 清洗/变换 -> 导出
  - 导入评论 -> 分词/停用词 -> 情感分析 -> 关键词/主题/词云 -> 导出

## 4. 推荐总体架构

### 4.1 架构总览

建议采用单机单用户高质量架构，避免引入过重中间件：

- 前端：Vue 3 + Vite + Ant Design Vue + Pinia + Axios + ECharts 5 + echarts-gl + echarts-wordcloud
- 后端：Python + FastAPI + Pydantic + SQLAlchemy/SQLModel
- 数据处理：Polars 优先，必要时 Pandas 兼容
- 机器学习：scikit-learn + xgboost + lightgbm
- 中文文本：jieba + SnowNLP
- 任务执行：ProcessPoolExecutor + 后端任务编排器
- 元数据存储：SQLite
- 数据快照与产物存储：本地文件系统 + Parquet/CSV/JSON/HTML/图片
- 实时进度：SSE

### 4.2 为什么这样设计

- 需求明确是自用系统，优先单机架构，减少部署复杂度。
- SQLite 适合元数据、任务记录、导出记录，但不适合直接承载百万级明细数据。
- 大数据实体更适合落在 Parquet 快照文件，利于复现、导出、缓存和高效读取。
- SSE 足以覆盖单向进度推送，复杂度低于 WebSocket。
- ProcessPoolExecutor 足以支持本地长任务，并保留后续迁移到 Celery/Redis 的空间。

## 5. 核心域模型

### 5.1 核心实体

- `Project`
- `Dataset`
- `Snapshot`
- `Pipeline`
- `Run`
- `Task`
- `Artifact`
- `Template`
- `Setting`
- `AuditLog`
- `Notification`

### 5.2 关键关系

- 一个项目包含多个数据集、流程、任务、导出物、模板和设置项。
- 一个数据集可以生成多个快照。
- 一个流程可以执行多次，形成多个 Run。
- 一个 Run 对应一个或多个 Task。
- 一个 Task 可以产出多个 Artifact。
- 所有核心实体都应带 `project_id`，保证隔离。

## 6. 存储方案

### 6.1 元数据存储

SQLite 存以下内容：

- 项目
- 数据集元信息
- 快照元信息
- 流程定义
- Run 记录
- Task 状态
- Artifact 索引
- 模板
- 设置
- 日志索引
- 通知

### 6.2 文件存储

建议目录结构：

```text
storage/
  projects/
    {project_id}/
      datasets/
        {dataset_id}/
          raw/
          working/
          snapshots/
      artifacts/
        charts/
        reports/
        html/
        exports/
      logs/
      cache/
      templates/
```

### 6.3 数据格式建议

- 原始导入保留原文件副本
- 工作副本优先使用 Parquet
- 导出按需求支持 CSV / Excel / JSON / YAML / PNG / SVG / HTML / PDF

## 9. 通用业务流程

### 9.1 数据导入流程

1. 用户选择项目
2. 用户上传文件
3. 后端创建导入任务
4. 解析文件并校验 schema
5. 保存原始文件
6. 生成 Dataset 与首个 Snapshot
7. 前端刷新数据表与任务状态

### 9.2 表格编辑流程

1. 前端加载工作副本分页数据
2. 用户编辑单元格
3. 后端记录变更并更新工作副本
4. 若需复现，提示创建新快照

### 9.3 流程运行

1. 前端提交流程参数
2. 后端绑定输入快照
3. 创建 Run 与 Task
4. 后端执行步骤并写日志
5. 产出结果和 Artifact
6. 前端通过 SSE 更新进度

### 9.4 导出流程

1. 用户在模块页或导出中心发起导出
2. 后端生成 Artifact
3. 导出记录进入导出中心
4. 可再次下载或按生命周期清理

## 11. 性能方案

### 11.1 前端性能

- 表格采用服务端分页与虚拟滚动
- 图表采用视口懒渲染
- 隐藏图表实例必须销毁释放资源
- 颜色采用回调，不传大批量颜色数组
- 导出时临时创建 SVG 实例
- 大导出、本地轻计算放入 Web Worker

### 11.2 后端性能

- 图表数据聚合优先返回
- 统计与建模支持抽样策略
- 大数据读取分块处理
- 同参数命中缓存
- 并发任务设上限
- 单任务内存和时长设阈值

## 12. 安全与风险控制

### 12.1 安全最小集

- 启动口令或本地登录
- 会话超时自动锁定
- 只读模式
- API Key 加密存储
- 导出报告中严禁明文暴露密钥

### 12.2 风险点

- 大数据内存爆涨
- 任务取消不彻底
- 3D 图表 GPU 资源泄漏
- 外部 API 成本失控
- 快照与工作副本混用导致不可复现

## 16. 结论

这不是一个只做页面或只做算法的项目，而是一套围绕“项目、数据、流程、任务、导出、复现、质量”构建的长期型分析工作台。实现时必须把基础设施和质量体系放到和业务功能同等重要的位置，否则后期一定会出现以下问题：

- 功能能演示但不能长期用
- 结果能生成但不能复现
- 页面好看但数据链路不稳定
- 跑得通一次但无法持续维护

因此后续所有实现必须围绕两个关键词推进：`完整性` 与 `质量`。
