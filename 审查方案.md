# 审查方案

## 1. 文档目标

本文档定义项目必须执行的审查机制。审查不是附属工作，而是项目能否交付的硬门槛。任何“功能大致能跑”但未经过系统性审查的版本，一律不得判定通过。

## 2. 审查总原则

- 至少进行 5 轮正式审查
- 每轮审查都必须有记录、有发现、有修复、有复验
- 审查对象包括需求覆盖、架构、交互、正确性、性能、安全、导出、任务机制、日志与可复现性
- 不允许把审查简化成“人工点几下页面看看”

## 3. 五轮强制审查

### 第一轮：需求覆盖审查

- 检查功能实现是否对齐 [需求文档_v2.md](D:\数据分析系统\需求文档_v2.md)
- 输出已覆盖项、缺失项、偏差项、暂缓项

### 第二轮：架构与数据链路审查

- 检查项目隔离、快照、Run、Task、Artifact 是否闭合
- 检查元数据与明细数据是否分离
- 检查接口命名与错误结构是否统一

### 第三轮：交互与页面一致性审查

- 检查白色简约拟态风格是否统一
- 检查三大模块布局是否一致
- 检查危险操作、错误提示、空状态、只读模式是否完整

### 第四轮：正确性、性能与安全审查

- 检查百万级分页和筛选是否流畅
- 检查图表是否走聚合与懒渲染
- 检查长任务是否可取消
- 检查外部 API 风险提示和密钥保护

### 第五轮：验收与回归审查

- 按最终验收清单做全链路复验
- 复验三条主闭环
- 复验任务、导出、日志、Run、快照联动

## 4. 专项审查清单

### 前端专项

- 设计系统是否统一
- 状态管理是否清晰
- 组件复用是否合理
- 是否滥用图表实例

### 后端专项

- 服务边界是否清晰
- 是否有未记录 Run/Task 的计算
- 错误结构是否统一
- 缓存与清理是否安全

### 联调专项

- SSE 是否稳定
- 任务状态是否同步
- 导出、日志、通知是否联动
- 前端是否依赖后端隐式字段

### 数据与算法专项

- 统计口径是否稳定
- 训练/验证切分是否可复现
- 是否存在数据泄漏
- 抽样是否有记录
- 模型指标与图表是否一致

## 5. 审查记录模板

每轮审查都应输出：

- 轮次
- 日期
- 审查范围
- 结论
- 问题列表
- 严重级别
- 修复状态
- 复验结果

## 6. 严重级别定义

- P0：阻断交付，功能错误或数据风险极高
- P1：必须修复，影响核心流程或复现性
- P2：建议本轮修复，影响体验或稳定性
- P3：可延后，但需入 backlog

## 7. 审查通过标准

- 5 轮审查全部完成
- P0 为 0
- P1 为 0
- 核心闭环全部通过
- 所有暂缓项有明确书面记录

## 8. 审查红线

- 用伪数据或静态 mock 冒充功能完成
- 故意跳过复杂需求，只保留展示流程
- 未实现可复现就宣称“分析完成”
- 任务失败不可追踪
- 为了赶进度忽略质量问题

## 9. 审查输出物

- 五轮审查记录
- 问题清单
- 风险摘要
- 发布阻断项列表
- 最终通过或不通过结论
